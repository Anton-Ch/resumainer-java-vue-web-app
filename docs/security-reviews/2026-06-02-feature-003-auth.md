### Risk Summary

This review covers Feature 003 at the design/planning stage — no Java or Vue code has been written yet. The overall posture is **MODERATE** because the design includes strong security foundations: BCrypt for passwords, PreparedStatement for SQL, rate limiting, session invalidation, generic auth errors, and HTTP-only cookies. However, several gaps exist that should be addressed before or during implementation: (1) password strength minimum of 8 chars is below current OWASP recommendations (12+), (2) CSRF protection is mentioned but not specified in implementation detail, (3) no Session Fixation prevention on login, (4) the "Remember me" 7-day session extends the attack surface without documented invalidation strategy.

The most impactful finding (SEC-001, HIGH) is the password policy — 8 characters with basic complexity is vulnerable to modern GPU-based brute force. This should be raised to 12+ characters in the spec before implementation begins, as changing it later will require updating both frontend Zod validation and backend Jakarta Validation annotations.

## Vulnerability Findings

### HIGH Finding

#### SEC-001: Weak Password Policy Below OWASP 2025 Recommendations

**Finding ID:** SEC-001
**Location:** `spec.md:214` (Assumptions), `contracts/api-contracts.md:25`
**OWASP Category:** A04:2025 — Cryptographic Failures
**CWE:** CWE-521 (Weak Password Requirements)
**CVSS Score:** 7.4 (High — CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N)

##### Description

The password policy requires minimum 8 characters with at least one uppercase, one lowercase, and one digit. Current OWASP ASVS (v4.0) and NIST SP 800-63B recommend minimum 12 characters for memorized secrets. An 8-character policy with limited character set is vulnerable to GPU-based brute force attacks (8-char NIST-compliant passwords can be cracked in under 24 hours on consumer GPUs with ~10^14 combinations vs ~10^17 for 12-char).

##### Affected Code

```
spec.md:214 — "Password strength requirements: minimum 8 characters"
contracts/api-contracts.md:25 — "password | Required, min 8 chars"
tasks.md T059 — PasswordStrengthValidator references 8 chars
```

##### Exploit Scenario

1. Attacker obtains password hash via SQL injection or DB leak
2. Modern GPU (RTX 4090) can attempt ~10 billion BCrypt hashes per hour
3. 8-char alphanumeric password space with minimal complexity ≈ 2.8 × 10^14 combinations
4. Estimated time to exhaust 50% of space: **14+ hours with consumer GPU**
5. 12+ char password: **~200+ years** equivalent

##### Impact

Account takeover of any user whose password is 8 characters. Especially impactful for admin accounts.

##### Remediation

1. Raise minimum password length to **12 characters** in all artifacts
2. Maintain existing complexity requirements (uppercase, lowercase, digit)
3. Consider adding special character requirement for bonus entropy
4. Update Zod schema in T049, Jakarta `@Size` in T026, and PasswordStrengthValidator in T059
5. Update contract in api-contracts.md

##### Fixed Code Example

```diff
- spec.md:214 — "minimum 8 characters, at least one uppercase letter, one lowercase letter, and one digit"
+ spec.md:214 — "minimum 12 characters, at least one uppercase letter, one lowercase letter, one digit, and one special character"

- api-contracts.md:25 — "password | Required, min 8 chars"
+ api-contracts.md:25 — "password | Required, min 12 chars, at least 1 uppercase, 1 lowercase, 1 digit, 1 special"
```

##### Spec-Kit Task

**TASK-SEC-001:** Raise password strength to OWASP 2025 standard (12+ chars, add special character)

- Severity: High
- Category: A04:2025 — Cryptographic Failures
- Location: spec.md:214, api-contracts.md:25, tasks.md T059
- Description: Update min password length from 8 to 12 chars, add special character requirement. Update Zod schema, Jakarta @Size annotation, and PasswordStrengthValidator.
- Acceptance Criteria: Registration rejects passwords < 12 chars. Unit tests verify new policy. Contracts updated.

### MEDIUM Findings

#### SEC-002: Session Fixation — No Explicit Session Regeneration on Login

**Finding ID:** SEC-002
**Location:** `spec.md:145` (FR-006), `contracts/api-contracts.md` (Session Management)
**OWASP Category:** A07:2025 — Authentication Failures
**CWE:** CWE-384 (Session Fixation)
**CVSS Score:** 5.3 (Medium)

##### Description

After successful login, the spec says "create user account, automatically sign in the user" but does NOT specify that the HttpSession must be regenerated (invalidate old + create new). Without session regeneration, an attacker who obtains a pre-authentication session ID (via XSS, network sniffing, or session prediction) will retain the same session ID after login, giving them authenticated access.

##### Affected Artifacts

- spec.md FR-006: "automatically sign in the user" — no session regeneration mention
- spec.md FR-013: "redirect the user to User Home" — no session regeneration mention
- contracts/api-contracts.md Session Management section: no mention of session regeneration

##### Remediation

Add session regeneration to login flow in AppInitializer or AuthController:
```java
// In AuthController.login():
HttpSession oldSession = request.getSession(false);
if (oldSession != null) {
    oldSession.invalidate();
}
HttpSession newSession = request.getSession(true);
// Set user attribute on newSession
```

Update FR-013 to: "Upon successful login, the system MUST invalidate any existing session and create a new authenticated session."

Update api-contracts.md Session Management to include session regeneration.

##### Spec-Kit Task

**TASK-SEC-002:** Add session regeneration on login to prevent session fixation

#### SEC-003: CSRF Protection Not Specified in Implementation Detail

**Finding ID:** SEC-003
**Location:** `spec.md:201` (Constitution V — mentions CSRF but no implementation detail)
**OWASP Category:** A01:2025 — Broken Access Control
**CWE:** CWE-352 (Cross-Site Request Forgery)
**CVSS Score:** 5.3 (Medium)

##### Description

The spec's Constitution section mentions "CSRF protection on auth forms" but no artifact (spec, plan, tasks, contracts) specifies HOW CSRF protection is implemented. Neither Spring Security's built-in CSRF filter (not available in pure Spring MVC without Boot) nor a custom token mechanism is documented. Without explicit CSRF protection, an attacker could trick an authenticated user into submitting registration/logout requests.

##### Affected Artifacts

- spec.md:201 — "CSRF protection on auth forms" (vague, no mechanism)
- plan.md: No CSRF section in Research or Technical Context
- tasks.md: No CSRF implementation task exists
- api-contracts.md: No CSRF token contract

##### Remediation

Add a CSRF section to plan.md Research. Options:
1. **Custom CSRF filter**: Generate token on login, store in session, include as header in Vue fetch calls, validate on backend via custom Filter/Interceptor
2. **Double-submit cookie pattern**: Set random cookie on login, Vue reads cookie and sends as header, backend validates match

For MVP simplicity, recommend Double-submit cookie: `CsrfTokenFilter` as a OncePerRequestFilter, token generated on `GET /api/auth/status` (authenticated=true response), validated on POST/PUT/DELETE.

Add Phase 5 task for CSRF filter implementation.

##### Spec-Kit Task

**TASK-SEC-003:** Implement CSRF protection for auth form submissions

#### SEC-004: "Remember Me" 7-Day Session Lacks Explicit Invalidation Strategy

**Finding ID:** SEC-004
**Location:** `spec.md:150` (FR-011)
**OWASP Category:** A07:2025 — Authentication Failures
**CWE:** CWE-613 (Insufficient Session Expiration)
**CVSS Score:** 4.0 (Medium)

##### Description

The "Remember me" feature extends session TTL to 7 days. The spec does not address:
- Can the remember-me session be explicitly invalidated (beyond logout)?
- Is there a maximum number of concurrent remember-me sessions?
- What happens after password change — are all existing remember-me sessions invalidated?

Without password-change invalidation, a user who changes password (after suspected compromise) leaves old 7-day sessions active.

##### Affected Artifacts

- spec.md FR-011: rememberMe extends to 7 days — no invalidation strategy
- tasks.md T061: "set session max inactive interval to 604800" — no invalidation on password change

##### Remediation

Add to spec.md or plan.md:
1. **Password change invalidation**: When `password_hash` is updated, invalidate ALL sessions for that user (except current one). Track session tokens mapped to user_id.
2. **Maximum concurrent sessions**: Cap at 5 per user. Oldest session invalidated when limit reached.
3. **Document in FR-011**: Add "Changing password invalidates all existing 'Remember me' sessions."

##### Spec-Kit Task

**TASK-SEC-004:** Add remember-me session invalidation on password change

#### SEC-005: AuthInterceptor Path Pattern May Overblock Future Public Endpoints

**Finding ID:** SEC-005
**Location:** `contracts/api-contracts.md:148` (Interceptor Paths)
**OWASP Category:** A06:2025 — Insecure Design
**CWE:** CWE-269 (Improper Privilege Management)
**CVSS Score:** 4.0 (Medium)

##### Description

The AuthInterceptor protects `/api/**` — this is intentionally broad for MVP. However, future features (public resume PDF access, Swagger/OpenAPI) will need paths excluded from auth. If these are added without updating the interceptor pattern, they will be blocked by auth, causing 401 errors that are hard to debug.

##### Affected Artifacts

- contracts/api-contracts.md:148 — `| /api/** | Yes | — |`

##### Remediation

Plan for future exclusion patterns now:
```java
// In WebConfig.addInterceptors():
registry.addInterceptor(authInterceptor)
    .addPathPatterns("/api/**")
    .excludePathPatterns(
        "/api/auth/**",
        "/api/public/**",    // Future: public resume PDF access
        "/api/swagger/**",   // Future: Swagger/OpenAPI
        "/api/health"        // Future: health check
    );
```

Document this in plan.md Technical Context or constraints.

##### Spec-Kit Task

**TASK-SEC-005:** Add future-proof exclusion paths to AuthInterceptor configuration

#### SEC-008: Missing Security Headers in Nginx Configuration

**Finding ID:** SEC-008
**Location:** `plan.md` Phase 3 (Docker & Integration)
**OWASP Category:** A02:2025 — Security Misconfiguration
**CWE:** CWE-693 (Protection Mechanism Failure)
**CVSS Score:** 3.1 (Low)

##### Description

The Nginx container is specified to serve the Vue SPA and proxy `/api/*` to Tomcat. No security headers are mentioned in any artifact. Missing headers include:
- `Content-Security-Policy` — prevents XSS
- `X-Frame-Options: DENY` — prevents clickjacking
- `Strict-Transport-Security` — enforces TLS

##### Remediation

Add to Nginx configuration in `plan.md` Phase 3 Step 3 or Dockerfile:
```nginx
add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data:; connect-src 'self'" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

**TASK-SEC-006:** Add security headers to Nginx configuration

### INFORMATIONAL Findings

#### SEC-009: BCrypt Cost Factor Not Specified

**Finding ID:** SEC-009
**Location:** `plan.md` Phase 1 Step 4 (PasswordService)
**OWASP Category:** A04:2025 — Cryptographic Failures
**CWE:** CWE-326 (Inadequate Encryption Strength)
**CVSS Score:** 0.0 (Informational)

##### Description

The spec/plan mandates BCrypt but does not specify the cost factor (work factor). Default BCrypt cost factor is 10 (2^10 = 1024 rounds). Current best practice for 2025 is cost factor 12+ for new implementations. As hardware improves, this should be reviewable via configuration.

##### Remediation

Add to plan.md Technical Context or PasswordService description:
- BCrypt cost factor: minimum 10 (default), configurable via `application.properties` as `security.bcrypt.cost=10`
- For production: use cost factor 12

**Verification**: Ensure `PasswordService` accepts configurable cost factor via constructor injection.

## Architecture Risks

### Risk Category: Trust Boundaries

#### Risk Description

The primary trust boundary is between Nginx (frontend) and Tomcat (backend). Both are on the same Docker network. No internal TLS between containers. An attacker who compromises the Nginx container could:
1. Access `/api/auth/*` internal endpoints (already unauthenticated — low risk)
2. Access the PostgreSQL container (requires DB credentials — mitigated by separate DB user)
3. Intercept/modify traffic between services (Docker internal network — low risk)

#### Risk Assessment

**Likelihood:** Low
**Impact:** Medium
**Risk Level:** Low

#### Mitigation Recommendations

- Use separate Docker networks for frontend↔backend and backend↔database
- DB user should have minimum required privileges (no DDL from application)
